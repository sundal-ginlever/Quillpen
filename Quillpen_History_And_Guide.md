# 📝 Quillpen (inkcanvas) 아카이브 가이드 및 개발 서사

Quillpen은 무한한 캔버스 위에 메모, 스케치, 스프레드시트, 이미지를 자유롭게 배치하고 연결할 수 있는 단일 페이지 웹 애플리케이션입니다.

---

## 📖 1. 프로젝트 개요 및 사용자 가이드

### 🚀 시작하기
1. **로컬 모드**: 브라우저에서 실행 시 기본적으로 로컬 모드로 동작하며, 데이터는 브라우저의 `localStorage`에 자동 저장됩니다.
2. **클라우드 모드**: Supabase와 연동하여 실시간 동기화와 클라우드 저장 기능을 활성화할 수 있습니다. (모든 데이터 테이블은 `q_` 접두사를 준수합니다)

### 🛠️ 지원되는 위젯
*   **📝 메모 (M)**: 텍스트 입력, 배경색 변경, 자유 리사이즈.
*   **🎨 스케치 (S)**: 자유 드로잉, 8색상/4굵기, 지우개 지원 (RDP 알고리즘 최적화).
*   **📊 스프레드시트 (T)**: 엑셀 호환 표 계산기. 다중 수식, 행/열 조절, 셀 서식 지원.
*   **🖼️ 이미지 (I)**: Base64 인코딩을 통한 파일 드래그 앤 드롭 업로드 지원.

### ⌨️ 필수 단축키
*   **내비게이션**: `V`(선택), `H`(이동), `Ctrl+휠`(줌인/아웃), `Ctrl+0`(줌 초기화), `Ctrl+Shift+F`(화면 맞춤)
*   **편집**: `Ctrl+Z`(실행 취소), `Ctrl+Y`(다시 실행), `Ctrl+D`(복제), `Ctrl+L`(잠금), `Delete`(삭제)
*   **시스템**: `Ctrl+F`(검색), `Ctrl+Shift+S`(공유), `Ctrl+Shift+E`(내보내기)

### 🧮 스프레드시트 수식
*   `=SUM(A1:C5)`, `=AVG(A1:A10)`, `=MIN(B1:B10)`, `=MAX(B1:B10)`
*   `=COUNT(A1:C5)`, `=COUNTA(A1:A10)`, `=PRODUCT(A1:A5)`
*   `=IF(A1>10, "크다", "작다")`, `=CONCAT(A1, "원")`

## 🏛️ 2. 개발 서사 (Development History)

이 프로젝트는 초기 **단일 HTML 파일 모노리스(Monolith) 아키텍처**로 시작되었습니다. 모든 뷰 로직과 비즈니스 로직, 상태 관리가 하나의 파일(`index.html`)에 응집되어 있었으나, 점차 기능이 방대해지면서 유지보수의 한계에 부딪혔습니다. 

이후, LLM AI 어시스턴트(Gemini 3.1 Pro 및 3 Flash)와의 협업을 통해 **ES Module 기반의 완전한 모듈형 아키텍처**로 프로젝트를 완전히 해체하고 재조립하는 여정을 거쳤습니다.

### 🏗️ Phase 1: 아키텍처 분리 및 긴급 버그 수정 (Hotfixes)
거대한 4,000줄의 코드를 기능별 모듈(`camera.js`, `sync.js`, 위젯 별도 파일 등)로 성공적으로 분리했습니다.
*   분리 과정에서 발생한 캔버스 밑으로 클릭이 투과되는 **"팬텀 위젯" 생성 버그**를 `isUI()` 가드 함수를 도입하여 해결했습니다.
*   스코프 문제로 고립된 **PWA 설치 트리거**와 **미니맵 로직**을 전역 접근이 가능하게 복구했습니다.

### ⚡ Phase 2: 렌더링 성능 최적화 (Performance)
모듈화 이후 발생한 렌더링 병목 현상을 해결했습니다.
*   **미니맵 Layout Thrashing 해결**: 매 프레임 DOM의 레이아웃을 읽는 대신 `ResizeObserver`를 사용하여 치수를 캐싱함으로써 프레임 드랍을 막았습니다.
*   **위젯 이동 GPU 가속**: 위젯 드래그 시 `top/left` 대신 `transform: translate3d`를 임시 적용하여 브라우저의 페인팅 파이프라인을 최적화하고 60fps를 확보했습니다.

### 🧩 Phase 3: 상태 관리 고도화 및 결합도 분리 (Architecture)
초기 모듈화 단계에서 사용되었던 임시 브릿지 객체(`window._appModules`)의 의존성을 해결했습니다.
*   **Event Bus (Pub/Sub) 도입**: `events.js`를 만들어 `camera:change`, `app:save`, `widget:render` 등의 이벤트를 중앙 제어하게 했습니다.
*   전체 코드베이스에서 `window._appModules` 호출을 100% 제거하고 완벽한 모듈 독립성을 확보했습니다.
*   `IndexedDB` 확장을 고려한 스토리지 어댑터(`storage` 객체)를 `sync.js`에 구축했습니다.

### 🛡️ Phase 4: 보안 및 데이터 무결성 (Security & Sync)
협업 환경을 위한 엔터프라이즈 수준의 안정성을 확보했습니다.
*   **XSS(크로스 사이트 스크립팅) 방어**: `utils.js`에 경량 `sanitize()` 함수를 구현하고, 캔버스 이름 및 UI 상태 등 `innerHTML`이 사용되는 영역에 강제 적용했습니다.
*   **Presence (실시간 사용자 가시성)**: Supabase Presence를 연동하여 실시간 접속 유저를 트래킹할 수 있는 기초 채널 구독 로직을 완성했습니다.

### 🚀 Phase 5: 고성능 캔버스 기반 엑셀 엔진 (Luckysheet) 도입
기존 DOM 기반 스프레드시트의 한계를 넘어서는 전문적인 데이터 처리 환경을 구축했습니다.
*   **Luckysheet 통합**: 수만 개의 셀도 지연 없이 처리하는 Canvas 기반의 Luckysheet 엔진을 성공적으로 통합했습니다. 복잡한 수식 엔진과 엑셀 스타일의 UI를 그대로 제공합니다.
*   **Sync 2.0 (Double Buffering)**: Luckysheet의 거대한 JSON 데이터(`luckyData`)를 유실 없이 전송하기 위해 `syncingChanges` 버퍼와 디바운스(Debounce) 로직을 도입했습니다. 네트워크 지연 중에도 데이터가 안전하게 보호되며 연속적으로 푸시됩니다.
*   **데이터 자가 치유(Sanitization)**: 앱 로드 시 비정상적인 데이터 구조를 감지하면 즉시 표준 그리드로 복구하는 보호 로직을 구현하여 무한 로딩 현상을 원천 차단했습니다.
*   **UI 인터랙션 가드**: 위젯 리사이즈 시 `setPointerCapture`를 통해 마우스 커서 이탈을 방지하고, 엑셀 편집 중 키보드 이벤트가 캔버스 단축키와 충돌하지 않도록 이벤트 전파 제어 로직을 완성했습니다.
*   **멀티 탭 동기화**: 하나의 위젯 안에서 여러 개의 시트(탭)를 생성하고 이를 통합하여 클라우드에 실시간 저장하는 구조를 확립했습니다.

### 📊 향후 계획: 차세대 멀티 인스턴스 엔진 (FortuneSheet) 전환
현재 Luckysheet의 기술적 한계(싱글톤 구조로 인한 멀티 위젯 제약)를 극복하기 위한 로드맵입니다.
*   **FortuneSheet 교체**: 하나의 캔버스에 여러 개의 스프레드시트 위젯을 동시에 띄울 수 있도록, 멀티 인스턴스를 지원하는 차세대 엔진인 FortuneSheet(Luckysheet의 후속작)로의 전환을 검토 중입니다.
*   **엑셀 호환성 강화**: 원본 엑셀 파일(`.xlsx`)을 위젯으로 직접 드래그 앤 드롭하여 불러오고, 수식과 서식을 유지한 채 내보내는 기능을 더욱 정교화할 예정입니다.

### 🏁 결론
거대하고 불안정했던 "하나의 파일"이, 확장성과 유지보수성이 극대화된 **안전하고 빠른 모듈형 웹 애플리케이션**으로 진화했습니다. 이는 성능, 아키텍처, 보안이라는 3박자를 모두 고려한 성공적인 리팩토링 사례로 남게 되었습니다.
